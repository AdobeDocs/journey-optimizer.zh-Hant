# 歷程與行銷活動：選擇正確的方法 {#journeys-vs-campaigns}

>[!BEGINSHADEBOX]

**在此頁面上：**&#x200B;將歷程與動作和API觸發的行銷活動進行比較，讓您能夠針對Adobe Journey Optimizer中的每個行銷使用案例選擇正確的方法。 如需協調的行銷活動，請參閱[開始使用協調的行銷活動](../orchestrated/gs-orchestrated-campaigns.md)。

>[!ENDSHADEBOX]

[!DNL Adobe Journey Optimizer]提供兩種主要方式來聯絡及吸引您的客戶：**歷程**&#x200B;和&#x200B;**行銷活動**。 歷程專為受客戶行為驅動的即時、多步驟協調而設計，而行銷活動則更適合對已定義的對象進行一次性或排程廣播，或是對邊緣的傳入頻道啟用以實現低延遲的個人化。

>[!NOTE]
>
>**協調的行銷活動**&#x200B;具有獨特的架構特性（中心端批次執行、多實體關聯式資料），需要專屬的指引。 為了避免過度簡化，它們不會包含在下面的比較中。 [進一步瞭解協調的行銷活動](../orchestrated/gs-orchestrated-campaigns.md)

## 您應該使用哪種方法？ {#decision-guide}

答案通常歸結為一個問題：*需要發生什麼，以及針對誰？*

如果您需要&#x200B;**每個客戶以自己的步調移動** — 根據他們的行為接收訊息、等待，然後回應他們的下一個動作 — 請使用&#x200B;**歷程**。 歷程會追蹤每個設定檔的位置及其完成的工作，使其非常適合多步驟體驗，例如上線順序、購物車放棄流程或忠誠度計畫。

如果您想要&#x200B;**依排程**&#x200B;傳送訊息給一組人員（電子報、產品公告、季節性促銷活動），請使用&#x200B;**動作行銷活動**。 對象中的每個人都會同時收到訊息，不需要每個設定檔的邏輯。 行動行銷活動也支援傳入頻道啟用（應用程式內、網路、內容卡片、程式碼型），以實現低延遲的邊緣個人化。

如果&#x200B;**外部系統需要立即觸發訊息** — 來自電子商務平台的訂單確認、來自您的物流系統的運送警示、來自您應用程式的密碼重設 — 針對單一隨選訊息使用&#x200B;**API觸發的行銷活動**，或如果該觸發需要啟動多步驟流程，則使用&#x200B;**單一事件歷程**。

若您需要具有進階分段、多實體資料或確切預先傳送計數的&#x200B;**複雜批次工作流程**，請參閱[開始使用協調的行銷活動](../orchestrated/gs-orchestrated-campaigns.md)。

>[!TIP]
>
>**不確定從何處開始？** 大部分的團隊會將歷程用於行為、事件導向的體驗，並將行動行銷活動用於排程通訊。 這兩者涵蓋大部分的使用案例。

## 兩者的比較方式 {#quick-overview}

| 方法 | 最適合 | 執行樣式 |
|----------|----------|-----------------|
| **歷程** | 具有條件式邏輯的多步驟、即時客戶體驗 | 1:1協調流程 — 每個設定檔都有自己的步調 |
| **動作行銷活動** | 已排程或循環啟用對象 | 批次執行 — 在傳送時一起處理的對象 |
| **API觸發的行銷活動** | 來自外部系統的事件導向或交易式訊息 | 隨選執行 — 由具有裝載的API呼叫觸發 |

## 每種方法的運作方式 {#key-distinctions}

### 歷程： 1:1即時協調

歷程是畫布，每個設定檔會依自己的速度行進自己的路徑。 AJO會追蹤每個人在流量中的位置，並即時對其行為做出反應，無論是他們採取的行動、閒置時間還是設定檔中的變更。

主要功能包括建立步驟之間個人化時機的等待活動、將設定檔路由至不同路徑的條件式分支、控制客戶接收訊息頻率的頻率限定，以及在上線前驗證邏輯的測試模式。 歷程也可以將設定檔分割為以百分比為基礎的群組，以便跨路徑進行A/B實驗。

放棄購物車歷程清楚說明了差異：

```
Customer A: Abandoned cart → Wait 2 hours → No purchase? → Send reminder → Purchased? → End
Customer B: Abandoned cart → Wait 2 hours → Already purchased → End immediately
```

每位客戶會根據實際作業體驗不同的時間表。 [進一步瞭解歷程](../building-journeys/journey.md)

### 行銷活動：批次或觸發傳送

行銷活動會執行單一動作 — 一次對對象中的每個人，或在外部系統呼叫時隨選執行。 沒有畫布和每個設定檔的狀態：所有設定檔的處理方式都相同。

**動作行銷活動**&#x200B;傳送至排程對象（一次性或週期性），也支援多表面傳入傳送 — 每個行銷活動最多10個傳入頻道動作，並根據對象成員資格或設定檔屬性建立訊息變體的鎖定目標規則。

**API觸發的行銷活動**&#x200B;會在外部系統呼叫API時立即引發，而訊息個人化則由該呼叫中傳送的裝載資料驅動。

```
Monday 9 AM → Send newsletter to 100,000 subscribers → All receive simultaneously
```

[進一步瞭解行銷活動](../campaigns/get-started-with-campaigns.md)

## 使用案例範例 {#use-cases}

| 使用案例 | 建議做法 | 原因 |
|----------|---------------------|-----|
| 歡迎新客戶採用多步驟上線 | 歷程 | 即時進入、多個接觸點、條件路徑 |
| 含提醒序列的購物車放棄 | 歷程 | 即時觸發器、等待時間、條件式後續追蹤 |
| 根據行為重新與非作用中使用者互動 | 歷程 | 由對象資格觸發，個人化路徑 |
| 業務事件觸發的閃購 | 歷程（業務事件） | 影響多個客戶的即時觸發器 |
| 每月給訂閱者的電子報 | 動作行銷活動 | 已排程傳送給對象的訊息 |
| 給所有客戶的促銷宣告 | 動作行銷活動 | 一次性訊息，立即傳遞 |
| 訂單確認或送貨警示 | API 觸發的行銷活動 | 外部系統觸發器，立即單次傳遞 |
| API觸發的多步驟流程 | 歷程（單一事件） | 外部系統透過API傳送事件；歷程協調後續步驟 |
| 具有多實體資料的複雜批次工作流程 | 協調的行銷活動 | 請參閱[開始使用協調的行銷活動](../orchestrated/gs-orchestrated-campaigns.md) |

## 常見問題 {#common-questions}

+++ 我可以在行銷策略中結合歷程和行銷活動嗎？

可以。 許多組織會針對不同情境使用所有方法：

* **行為即時參與歷程**
* 針對排程通訊或傳入啟用的&#x200B;**動作行銷活動**
* 異動訊息的&#x200B;**API觸發的行銷活動**
* 針對複雜、資料密集的批次行銷活動，**協調的行銷活動** — 請參閱[開始使用協調的行銷活動](../orchestrated/gs-orchestrated-campaigns.md)

針對每個使用案例使用正確的工具，而不是針對所有內容強制使用一種方法。

+++

+++ 我可以將行銷活動轉換為歷程嗎，反之亦然？

否，您必須以適當格式重建體驗。 不過，您可以重複使用內容、對象和邏輯概念。

+++

+++ 哪種方式最容易建置？

動作行銷活動通常是最簡單（提供給對象的單一接觸點），隨後是API觸發的行銷活動，然後是歷程，由於多步驟邏輯，需要更多設計工作。

+++

+++ 哪些縮放比例適合大型受眾？

這三種模式都可以很好的擴展；正確的選擇取決於您的模式：

* **讀取對象歷程**&#x200B;和&#x200B;**動作行銷活動**&#x200B;已針對大量批次對象最佳化。
* **單一（事件型）歷程**&#x200B;會在事件發生時個別處理設定檔，因此規模取決於事件數量和輸送量。

如需具有大型資料集和多實體資料的複雜細分，請參閱[協調的行銷活動](../orchestrated/gs-orchestrated-campaigns.md)。

+++

+++ 我可以在歷程和行銷活動中使用相同的對象嗎？

可以。 在[!DNL Adobe Experience Platform]中建立的對象可用於歷程、動作行銷活動和協調的行銷活動。 API觸發的行銷活動是裝載導向，不使用相同方式的預先建立對象。

**意圖：**
* 瞭解歷程、動作行銷活動和API觸發的行銷活動之間的主要差異
* 使用決定指南和比較表，針對指定的行銷使用案例選取正確的方法
* 瞭解動作行銷活動何時支援傳入頻道啟用與傳出廣播

* 在行銷策略中有效結合多種方法

**字彙表：**
* **歷程**：多步驟的即時協調流程，每個設定檔都會根據行為和事件以自己的步調前進。 *（產品特定）*
* **動作行銷活動**：將排程或週期性啟用傳送給對象的行銷活動 — 將傳出廣播或傳入頻道啟用傳送給邊緣，以進行低延遲的個人化。 *（產品特定）*
* **API觸發的行銷活動**：外部系統透過API呼叫所起始的行銷活動，以裝載驅動的個人化傳遞單一隨選訊息。 *（產品特定）*
* **協調的行銷活動**：支援多實體關聯資料、臨機對象構成和同盟資料來源的中心端批次行銷活動；此頁面上的比較表格未涵蓋。 *（產品特定）*
* **單一事件歷程**：單一設定檔動作即時觸發的歷程；在API傳送事件後需要多步驟協調時使用。 *（產品特定）*
* **傳入頻道啟用**：將個人化體驗傳遞至邊緣（程式碼型體驗、應用程式內、內容卡、網頁），以進行低延遲轉譯，並受「動作」行銷活動支援。 *（產品特定）*

**護欄：**
* 每個動作行銷活動最多10個傳入頻道動作（硬性限制） — 僅適用於傳入頻道：程式碼型體驗、應用程式內、內容卡、網路
* 協調的行銷活動會從此頁面的比較表中排除，以避免過度簡化；如需架構詳細資訊，請參閱專屬的協調行銷活動檔案

**術語：**
* 正式名稱：動作行銷活動 — 變體：「排程行銷活動」、「廣播行銷活動」
* 正式名稱：API觸發的行銷活動 — 變體：「交易式行銷活動」、「事件導向行銷活動」
* 請勿混淆：「動作行銷活動」（已排程/傳入傳送至對象）≠「API觸發的行銷活動」（隨選、裝載導向、無預先建立對象）≠「協調的行銷活動」（具有關聯式資料的中心端批次）
* 請勿混淆：「單一事件歷程」（由設定檔的即時動作觸發）≠「業務事件歷程」（由透過內部「讀取對象」步驟影響多個人的非設定檔事件觸發）
* 同義字：「傳入頻道啟用」=「傳入頻道動作」（在此頁面上可交換地用於「動作」行銷活動中邊緣傳送的體驗）

**常見問題集：**
* **問：何時應使用歷程而非動作行銷活動？**  — 當客戶需要跨多個接觸點以自己的步調進行即時條件式邏輯移動時，請使用歷程；使用動作行銷活動將排程或傳入的傳送給預先定義的對象。
* **問：動作行銷活動可以傳送到傳入頻道嗎？**  — 是。 動作行銷活動支援到邊緣的輸入管道啟用（程式碼型體驗、應用程式內、內容卡、網頁）以實現低延遲的個人化，每個行銷活動最多10個輸入動作，以及訊息變體的目標定位規則。
* **問：協調的行銷活動與動作行銷活動有何不同？**  — 協調的行銷活動會使用多實體關聯資料、精確的預先傳送計數、臨機對象構成和同盟資料支援來執行中心端批次執行；動作行銷活動是無狀態的單一執行傳送給Experience Platform對象。
* **問：何時應該使用API觸發的行銷活動與單一事件歷程？**  — 當外部系統需要立即觸發包含裝載資料的單一訊息時，使用API觸發的行銷活動；當在API傳送事件後需要多步驟協調時，使用單一事件歷程。
* **問：我可以在相同的行銷策略中結合歷程和行銷活動嗎？**  — 是。 使用歷程進行行為即時參與、使用動作行銷活動進行排程廣播或傳入啟用、使用API觸發的行銷活動進行異動訊息，以及使用協調的行銷活動進行複雜的批次工作流程。
