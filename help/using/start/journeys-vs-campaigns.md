# 歷程與行銷活動：選擇正確的方法 {#journeys-vs-campaigns}

Adobe Journey Optimizer提供三種強大的方式觸及及吸引客戶。 瞭解何時使用每個報表套裝，是建立有效行銷體驗的關鍵。

本指南可協助您根據特定行銷需求，在&#x200B;**歷程**、**動作行銷活動**、**API觸發的行銷活動**&#x200B;和&#x200B;**協調的行銷活動**&#x200B;之間選擇。

## 快速比較概述 {#quick-overview}

| 方法 | 最適合 | 執行樣式 |
|----------|----------|-----------------|
| **歷程** | 具有條件式邏輯的多步驟、即時客戶體驗 | 1:1協調流程 — 每個設定檔都有各自的步調 |
| **動作行銷活動** | 排程或循環播放至對象 | 批次執行 — 在傳送時一起處理的對象 |
| **API觸發的行銷活動** | 來自外部系統的事件導向或交易式訊息 | 隨選執行 — 由具有裝載的API呼叫觸發 |
| **協調的行銷活動** | 具有多實體分段的複雜批次工作流程 | 批次畫布 — 所有設定檔一起處理 |

## 詳細比較 {#detailed-comparison}

使用這份全方位的表格來瞭解主要差異：

| 功能 | 歷程 | 動作行銷活動 | API觸發的行銷活動 | 協調的行銷活動 |
|---------|----------|------------------|------------------------|----------------------|
| **主要用途** | 使用即時客戶內容進行多步驟1:1協調 | 傳送給對象的一次性或循環訊息傳送 | 由外部系統初始的交易式或事件導向訊息 | 具有複雜分段工作流程的多步驟批次行銷活動 |
| **畫布型別** | 1:1畫布 — 每個設定檔都會以自己的速度移動 | 無畫布 — 單一動作執行 | 無畫布 — 單一動作執行 | 批次畫布 — 所有設定檔一起處理 |
| **執行流程** | 循序動作，設定檔會維護整個歷程的狀態 | 同時執行給整個對象 | 每個API呼叫立即執行 | 具有活動和轉變的多步驟批次工作流程 |
| **進入機制** | 事件、受眾、資格、業務活動 | 手動啟動和排程 | 來自外部系統的API呼叫 | 已排程執行批次工作流程 |
| **資料模型** | 即時設定檔+事件資料 | 來自Experience Platform對象的設定檔資料 | API裝載資料（含選用的設定檔查閱） | 多實體關聯式資料（設定檔、產品、商店、預訂） |
| **區段** | 預先建立的對象+即時條件 | Experience Platform的預先建立對象 | 裝載導向鎖定目標（無排程對象） | 在包含精確計數的畫布中建立的隨選受眾 |
| **設定檔處理中** | 個人即時（當事件發生時） | 批次，一次完成 | 每個API呼叫，裝載導向 | 批次，以及多實體支援 |
| **個人化** | 即時內容資料+設定檔屬性 | 輪廓屬性 | 裝載資料+可選的設定檔屬性 | 精準定位的多實體資料 |
| **複雜性** | 具有分支、等待時間、條件的多步驟 | 單一動作或簡單工作流程 | 具有裝載對應的單一動作 | 具有分段、擴充、分割的多步驟批次工作流程 |
| **最適合** | 客戶生命週期歷程、入門、購物車放棄 | 促銷活動、電子報、公告 | 訂單確認、送貨警示、密碼重設 | 複雜的季節性行銷活動、多步驟促銷活動、產品推出 |
| **計時** | 連續，發佈後一律有效 | 排定的開始/結束日期 | 透過API的隨選、事件導向 | 依排程批次執行 |
| **狀態管理** | 維護即時動作的客戶狀態 | 無狀態執行 | 每個呼叫的無狀態執行 | 使用工作表進行批次處理 |
| **使用時機** | 需要即時決策邏輯的多個接觸點 | 在特定時間傳送給對象的簡單訊息 | 外部系統需要立即觸發訊息 | 需要複雜的分段、多實體資料或精確的預先傳送計數 |
| **獨特功能** | 即時回應、等待活動、設定檔型步調 | 排程，對象目標定位，速率控制 | API裝載對應，系統到系統觸發 | 關聯式資料集、多實體分段、精確計數、多級傳送 |

## 決定指南 {#decision-guide}

請依照此決策樹來選擇正確的方法：

### 步驟1：您的執行需求為何？

**個人對客戶行為的即時回應？**
→ **使用歷程**
* 設定檔需要按照自己的速度移動
* 以行為為基礎的條件式邏輯
* 即時上下文至關重要

**在排程的時間將簡單的訊息傳送給對象？**
→ **使用動作行銷活動**
* 所有設定檔都會同時接收訊息
* 排程或週期性傳送
* 不需要複雜的多步驟邏輯

**外部系統觸發立即訊息？**
→ **使用API觸發的行銷活動**
* 透過API呼叫隨選觸發
* 裝載驅動的個人化
* 不需要複雜的多步驟邏輯

**具有進階分段的複雜批次工作流程？**
→ **使用協調的行銷活動**
* 需要多實體資料（產品、商店、預訂）
* 需要精確的預先傳送計數
* 具有分割和擴充的多步驟批次處理

### 步驟2：驗證您的選擇

| 您的需求 | 建議做法 | 原因 |
|-----------|---------------------|-----|
| 歡迎新客戶採用多步驟上線 | 歷程 | 即時進入、多個接觸點、條件路徑 |
| 傳送每月電子報給訂閱者 | 動作行銷活動 | 傳送給對象的簡單排程訊息 |
| 含提醒序列的購物車放棄 | 歷程 | 即時觸發器、等待時間、條件式後續追蹤 |
| 給所有客戶的促銷宣告 | 動作行銷活動 | 一次性訊息，立即傳遞 |
| 根據行為重新與非作用中使用者互動 | 歷程 | 由對象資格觸發，個人化路徑 |
| 業務事件觸發的閃購 | 歷程（業務事件） | 影響多個客戶的即時觸發器 |
| 透過產品目錄整合進行季節性促銷活動 | 協調的行銷活動 | 多實體資料、複雜分段、確切計數 |
| API觸發的交易式訊息 | API觸發的行銷活動 | 外部系統觸發器，立即傳遞 |
| 每次預訂的多層級傳送 | 協調的行銷活動 | 多實體關係，每個預訂一則訊息 |

## 主要區別說明 {#key-distinctions}

### 歷程： 1:1即時協調

**其獨特之處：**
* 每個設定檔都會維護個別的狀態和內容
* 設定檔會以自己的步調進入和進度
* 根據行為和事件進行即時決策
* 等待活動建立個人化時間
* 條件式分支會為每個設定檔建立唯一路徑

**範例流程：**

```
Customer A: Abandoned cart → Wait 2 hours → No purchase? → Send reminder → Purchased? → End
Customer B: Abandoned cart → Wait 2 hours → Already purchased → End immediately
```

每個客戶都會根據其動作，體驗各自的歷程時間表。

[進一步瞭解歷程](../building-journeys/journey.md)

### 行銷活動：簡單批次或觸發式傳送

**其獨特之處：**
* 所有設定檔都以相同方式同時處理
* 無狀態執行 — 未維護任何內容
* 簡單排程或API觸發
* 廣播通訊的理想選擇

**範例流程：**

```
Monday 9 AM → Send newsletter to 100,000 subscribers → All receive simultaneously
```

每個人都同時收到相同的訊息。

**型別：**
* **動作行銷活動**：已排程傳送給對象（一次性或週期性）
* **API觸發的行銷活動**：由包含裝載資料的API呼叫觸發的隨選傳送

[進一步瞭解行銷活動](../campaigns/get-started-with-campaigns.md)

### 協調的行銷活動：批次畫布工作流程

**其獨特之處：**
* 具有活動和轉變的批次畫布（類似於歷程畫布，但以批次導向）
* 多實體關聯式資料支援（設定檔+產品+商店+預訂）
* 在畫布中建立隨選受眾
* 傳送前的準確計數（傳送前可見度）
* 多級傳送（每個實體一則訊息，例如，每個預訂）
* 所有設定檔一起批次處理

**範例流程：**

```
Query customers → Filter by purchase history → Split by region → 
Enrich with product data → Build segments → Send personalized offers → All in one batch execution
```

結合工作流程複雜性與批次行銷活動執行。

[進一步瞭解協調的行銷活動](../orchestrated/gs-orchestrated-campaigns.md)

## 使用案例範例 {#use-cases}

### 歷程使用案例

* **購物車放棄復原**：由購物車新增事件觸發，等待結帳，若未購買則傳送提醒
* **客戶上線**：根據設定檔資料提供個人化內容的多步驟歡迎系列
* **忠誠度層級升級**：當客戶達到新層級時觸發，傳送祝賀與福利
* **生日行銷活動**：以生日為基礎的專案、個人化優惠
* **重新參與**：由對象資格（非使用中）、漸進式外展所觸發

### Campaign使用案例（動作和API觸發）

**動作行銷活動：**
* **每月電子報**：已排程批次傳遞至訂閱者區段
* **促銷宣告**：對目標對象的時效性優惠方案
* **產品啟動**：協調所有客戶的公告
* **季節性問候語**：特定日期的假日訊息

**API觸發的行銷活動：**
* **訂單確認**：購買後由電子商務系統觸發
* **送貨通知**：由物流系統觸發
* **帳戶警示**：欺詐偵測系統觸發
* **密碼重設**：由應用程式中的使用者動作觸發

### 協調的行銷活動使用案例

* **包含目錄整合的季節性促銷活動**：查詢產品目錄、識別合格客戶、依偏好設定分段、傳送個人化產品推薦
* **特定商店的行銷活動**：使用商店詳細目錄資料鎖定接近特定商店地點的客戶
* **多重預約通訊**：每次預約傳送一則訊息（飯店預訂、航班預訂）
* **複雜的區段協調**：從多個資料來源擴充以逐步建置對象
* **預先傳送驗證**：在啟動主要行銷活動之前取得收件者的確切計數

## 常見問題 {#common-questions}

+++ 我可以在行銷策略中結合歷程和行銷活動嗎？

當然可以！ 大多陣列織會針對不同情境使用所有三種方法：

* 行為、即時參與的歷程
* 排程廣播通訊的行動行銷活動
* 異動訊息的API觸發行銷活動
* 針對複雜、資料密集的批次行銷活動，協調行銷活動

+++

+++ 我可以將行銷活動轉換為歷程嗎，反之亦然？

否，您必須以適當格式重建體驗。 不過，您可以重複使用內容、對象和邏輯概念。

+++

+++ 哪種方法比較容易建置？

動作行銷活動通常是最簡單的（傳送給對象的單一訊息），其次是API觸發的行銷活動、歷程（使用多步驟邏輯更複雜）和協調的行銷活動（由於畫布工作流程和多實體功能而最複雜）。

+++

+++ 哪些縮放比例適合大型受眾？

這三者都可以很好的擴展，但是：

* **讀取對象歷程**&#x200B;和&#x200B;**動作行銷活動**&#x200B;已針對大型批次對象最佳化
* **協調的行銷活動**&#x200B;在包含大型資料集的複雜細分方面表現卓越
* **單一歷程**&#x200B;會個別處理設定檔，因此規模取決於事件數量

+++

+++ 我可以在歷程和行銷活動中使用相同的對象嗎？

可以，在Adobe Experience Platform中建立的對象可用於所有三種方法。

+++
