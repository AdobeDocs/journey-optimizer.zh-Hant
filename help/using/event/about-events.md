# 使用歷程事件 {#about-events}

>[!BEGINSHADEBOX]

**在此頁面上：**&#x200B;瞭解三種事件型別、其結構描述需求、金鑰限制，以及如何為您的使用案例選擇正確的事件型別。

>[!ENDSHADEBOX]

>[!CONTEXTUALHELP]
>id="ajo_journey_event_list"
>title="歷程事件"
>abstract="Journey Optimizer 支援歷程中的三種事件類型：與特定個人行為連結的單元事件 (例如購買或忠誠度里程碑)；由全域事件觸發的業務事件 (例如航班取消或庫存更新)；以及在輪廓進入或退出某個客群時觸發的客群資格事件。 使用事件來觸發歷程，並針對您的輪廓協調正確的動作。"

使用事件來個別觸發歷程，當每個使用者進入歷程時傳送即時訊息給他們。 在事件設定中，您會設定歷程中預期的事件。 您可以使用多個事件 (在歷程的不同步驟中)，而好幾個歷程可以使用同一個事件。

事件設定是&#x200B;**強制**，必須由資料工程師執行。

>[!IMPORTANT]
>
>* 在設定事件之前，請確定您具有： **Journey Optimizer管理員**&#x200B;或&#x200B;**資料工程師**&#x200B;角色、已啟用&#x200B;**即時客戶個人檔案**&#x200B;的XDM結構描述、作用中的串流端點，以及正確沙箱的存取權。
>
>* 如需事件需求和限制（串流、查詢服務、批次擷取），請參閱[歷程護欄 — 事件](../start/guardrails.md#events-g)。

**執行什麼動作：**

| 任務 | 角色 |
| --- | --- |
| 設計和建立XDM結構描述 | 資料工程師 |
| 設定串流端點 | 資料工程師 |
| 設定單一和商務事件（**管理>事件**） | 資料工程師或管理員 |
| 在歷程中使用事件 | 歷程從業者 |
| 設定對象資格事件（在畫布上選取） | 歷程從業者 |

您可以設定三種型別的事件： **單一事件**、**商業事件**&#x200B;和&#x200B;**對象資格事件**。

## 單一事件 {#unitary-events}

**單一**&#x200B;事件已連結至人員。 它與人的行為相關（例如，某人購買產品、造訪商店、離開網站等等） 或是某人發生的事 (例如，某人達到 10,000 點忠誠點數)。 這是 [!DNL Journey Optimizer] 在歷程中會監聽的事件，以便協調下一個最佳動作。 單一事件可以是規則型或系統產生。 [瞭解如何建立單一事件](../event/about-creating.md)。

**結構描述需求：**&#x200B;已啟用XDM ExperienceEvent結構描述，該結構描述具有以人員為基礎的主要身分和&#x200B;**即時客戶設定檔**。

**範例：**&#x200B;客戶將專案新增至購物車並關閉瀏覽器。 系統會觸發購物車放棄的事件、設定檔即時進入歷程，並於一小時後收到復原電子郵件。

>[!NOTE]
>
>單一歷程包含重新進入護欄：預設會在歷程觸發後5分鐘內封鎖設定檔重新進入。 例如，如果某個事件在12:01觸發設定檔的歷程，而另一個事件在12:03到達，則該歷程不會重新開始該設定檔。

## 商業事件 {#business-events}

**企業**&#x200B;事件未連結至特定設定檔。 例如，可以是新聞警報、運動更新、航班變更或取消、詳細目錄更新、天氣事件等。雖然這些事件不是設定檔所特有，但任何數量的設定檔都可能有興趣：訂閱特定新聞主題的個人、航班上的乘客、對無存貨產品感興趣的購物者等。業務事件一律以規則為基礎。 當您在歷程中放置商業活動時，它會在之後自動新增&#x200B;**讀取對象**&#x200B;活動。 [瞭解如何建立商務活動](../event/about-creating-business.md)。

**結構描述需求：**&#x200B;具有非人員主要身分的時間序列XDM結構描述，且已填入`_id`和`timestamp`欄位。 針對最多1小時的對象匯出延遲時間規劃15分鐘。

**範例：**&#x200B;航空公司取消航班。 業務事件會引發，[!DNL Journey Optimizer]會讀取受影響乘客的對象，並傳送每個重新預約通知。

## 對象資格鑑定事件 {#audience-qualification-events}

當設定檔進入或退出對象時，會觸發&#x200B;**對象資格**&#x200B;事件。 例如，超過忠誠度支出閾值的客戶進入金級對象 — 該資格會即時觸發該設定檔的歷程（適用於串流對象）或在下一次批次評估。 與單一事件不同，對象資格可讓您使用對象定義的完整功能建立複雜的觸發邏輯，而不需要變更實施即可傳送新事件。 深入瞭解[對象資格事件](../building-journeys/audience-qualification-events.md)。

**結構描述需求：**&#x200B;不需要其他結構描述 — 事件有賴於Adobe Experience Platform中已建置的現有對象定義。

**範例：**&#x200B;客戶的忠誠度支出超過金級閾值。 他們的設定檔符合Gold對象的資格，歷程會自動觸發，並傳送歡迎獎勵。

>[!NOTE]
>
>未在&#x200B;**管理>事件**&#x200B;中設定對象資格事件 — 它們會直接在歷程畫布上選取作為歷程的第一步。

## 事件型別總覽 {#event-comparison}

| | 單一事件 | 業務事件 | 對象資格事件 |
| --- | --- | --- | --- |
| **連結到設定檔？** | 是 — 由特定個人的動作觸發。 | 否 — 由不與一個人員關聯的外部發生次數觸發。 | 是 — 當設定檔進入或退出對象時觸發。 |
| **專案行為** | 一個設定檔即時進入歷程。 | 透過自動讀取對象步驟輸入多個設定檔。 | 對象會籍變更時，會進入一個設定檔。 |
| **典型使用案例** | 購物車放棄復原、表單提交、應用程式登入、忠誠度里程碑。 | 航班取消、補充存貨警示、突發新聞、天氣事件。 | 重新參與失效客戶、忠誠度等級變更、VIP離站流程。 |
| **如何開始歷程** | 事件型登入 — 不需要對象。 | 業務事件+自動讀取對象（由Journey Optimizer新增）。 | 設定檔會進入或退出已定義的對象。 |
| 每個歷程&#x200B;**多次？** | 是 — 您可以跨歷程步驟聆聽數個單一事件。 | 否 — 每個歷程只有一個業務事件，放在開頭。 | 是 — 可與其他活動結合。 |
| **事件ID型別** | 規則型或系統產生。 | 一律以規則為基礎。 | 無事件ID — 根據對象成員資格評估而定。 |
| **已在Administration中設定？** | 是 | 是 | 否 — 直接在歷程畫布上選取。 |

>[!NOTE]
>
>歷程只能包含&#x200B;**一個**&#x200B;商業事件，這必須是第一個活動。 Journey Optimizer會自動在之後新增&#x200B;**讀取對象**&#x200B;活動，以定義哪些設定檔會接收該事件觸發的歷程。

## 事件ID型別 {#event-id-type}

對於&#x200B;**business**&#x200B;事件，事件ID型別一律以規則為基礎。

針對&#x200B;**單一**&#x200B;事件，有兩種事件識別碼：

* **規則型** 事件：此類型的事件不會產生 eventID。 使用簡單運算式編輯器，您只需定義一個規則，系統會使用該規則來識別將觸發您歷程的相關事件。 此規則可以根據事件裝載中可用的任何欄位，例如輪廓的位置或新增至輪廓購物車的項目數。

  >[!CAUTION]
  >
  >已為規則型事件定義上限規則。 對於指定組織，歷程可處理的合格事件數限製為每秒5,000個。 它對應於Journey Optimizer SLA。 請參閱您的Journey Optimizer授權和[Journey Optimizer產品說明](https://helpx.adobe.com/tw/legal/product-descriptions/adobe-journey-optimizer.html){target="_blank"}。

* **系統產生的** 事件：這些事件需要 eventID。 建立事件時，會自動產生此 eventID 欄位。 推播事件的系統不應產生 ID，而應傳遞有效裝載預覽中可用的 ID。

>[!NOTE]
>
>只有串流事件才能觸發歷程。 下列&#x200B;**無法**&#x200B;用來觸發歷程：
>
>* 批次擷取的事件
>* 透過&#x200B;**查詢服務**&#x200B;插入的事件
>* 來自內部[!DNL Journey Optimizer]資料集的事件（訊息回饋、電子郵件追蹤等）
>
>如果您無法取得串流事件，請根據這些事件建立對象，然後改用&#x200B;**讀取對象**&#x200B;活動。 若要僅將事件用於分段，請為&#x200B;**即時客戶個人檔案**&#x200B;啟用資料集。

## 如何選擇 {#choose-event-type}

使用以下條件為您的歷程選取正確的事件型別 — 關鍵問題是： **您是為特定人員觸發動作，還是廣播至多個設定檔？** [進一步瞭解歷程型別](../building-journeys/journey.md#journey-types)。

每個事件型別都會對應至特定歷程模式：

| 事件型別 | 歷程模式 |
| --- | --- |
| 單一事件 | 即時、單一設定檔歷程 — 在人員採取行動時立即觸發 |
| 業務事件 | 廣播歷程 — 透過自動讀取對象步驟鎖定許多設定檔 |
| 對象資格事件 | 區段觸發的歷程 — 設定檔進入或退出對象時引發 |

* **當觸發程式繫結至特定個人時（例如，購買、表單提交或忠誠度里程碑），請選擇單一事件**。 單一事件需要在結構描述中以人物為基礎的主要身分識別，並立即為該設定檔開始歷程。 [瞭解如何設定單一事件](../event/about-creating.md)。

* **當觸發程式是全球事件（例如產品重新上架、價格下降或航班取消）且您想要廣播到與該訊號相關的一組設定檔時，請選擇業務事件**。 業務事件必須是歷程的第一步，並透過&#x200B;**讀取對象**&#x200B;活動自動鎖定設定檔。 它們需要具有非人員主要身分以及`_id`和`timestamp`欄位的時間序列結構描述。 針對最多1小時的對象匯出延遲時間規劃15分鐘。 [瞭解如何設定業務事件](../event/about-creating-business.md)。

* **當觸發因素是進入或退出對象的設定檔，而您需要比單一事件所能提供的更複雜的分段邏輯時（例如，重新吸引剛達到支出臨界值的失效客戶，或當VIP成員退出忠誠度層級時觸發離任流程），請選擇對象資格事件**。 [進一步瞭解對象資格事件](../building-journeys/audience-qualification-events.md)。

>[!CAUTION]
>
>業務事件不能與單一事件或對象資格活動用於同一個歷程。

## 關鍵限制 {#key-constraints}

使用此摘要，在設定事件之前規劃實施作業。

| 限制 | 詳細資料 |
| --- | --- |
| 輸送量限制 | 所有沙箱（單一和讀取對象歷程）中，每個組織每秒5,000個事件 |
| 重新進入區塊 | 單一歷程觸發後，設定檔重新進入遭封鎖5分鐘 |
| 每個歷程的業務事件 | 最多1個，必須是第一個步驟 |
| 在一個歷程中混合業務+單一 | 不支援 |
| 批次事件 | 無法觸發歷程 — 請改用&#x200B;**讀取對象**&#x200B;活動 |
| 對象資格 — 管理 | 未在&#x200B;**管理>事件**&#x200B;中設定 — 直接在歷程畫布上選取 |
| 編輯即時活動 | 只能變更名稱、說明或新增裝載欄位 |

## 事件如何觸及Journey Optimizer {#data-cycle}

事件必須透過[Adobe Experience Platform串流擷取API](https://experienceleague.adobe.com/docs/experience-platform/xdm/api/getting-started.html?lang=zh-Hant){target="_blank"}以POST呼叫的形式傳送至[!DNL Journey Optimizer]。 承載必須遵循XDM格式，且事件結構描述必須啟用&#x200B;**即時客戶個人檔案**。

同時支援驗證和未驗證的串流模式。 來自內部[!DNL Journey Optimizer]資料集的批次擷取事件和事件（訊息回饋、電子郵件追蹤等）無法用於觸發歷程 — 請針對這些使用案例改用&#x200B;**讀取對象**&#x200B;活動。


## 事件輸送量限制 {#event-throughput}

Adobe Journey Optimizer會在組織層級，對所有沙箱針對每個事件型別實施個別的輸送量限制：

* **單一事件**：每秒5,000個事件
* **讀取以對象為基礎的歷程事件**：每秒5,000個事件

這些限制適用於使用中歷程中使用的所有事件，包括&#x200B;**即時**、**演習**、**已關閉**&#x200B;和&#x200B;**已暫停**&#x200B;歷程。 當達到限制時，新事件會排入佇列並以每秒5,000個的速度處理，直到佇列完全排清為止。

如需歷程處理率以及不同歷程型別如何影響輸送量的詳細資訊，[進一步瞭解歷程處理率](../building-journeys/entry-management.md#journey-processing-rate)。

下列型別的事件計入這些配額：

* **外部單一事件**：包含規則型事件和系統產生的事件。 如果相同的原始事件符合多個規則定義的資格，則每個相符的規則都會計為配額的個別事件。

* **對象資格事件**：如果在多個歷程中使用相同的串流對象，則每個使用量會個別計算。 例如，在兩個歷程的對象資格活動中使用相同對象時，會產生兩個計入的事件。

* **回應事件**：設定檔回應觸發的事件（已開啟電子郵件、已點按電子郵件等） 歷程中。

* **商業事件**：事件未繫結至特定設定檔，而是繫結至商業相關事件。

* **Analytics事件**：如果已啟用[與Adobe Analytics的整合以觸發歷程](about-analytics.md)，則也會包含這些事件。

* **繼續事件**：當設定檔從暫停的歷程繼續時觸發的技術事件。 深入瞭解[繼續暫停的歷程](../building-journeys/journey-pause.md#journey-resume-steps)。

* **等待節點完成事件**：當設定檔退出等待節點時，會產生技術事件以繼續歷程。

>[!NOTE]
>
>除了等待和恢復事件，根據讀取對象在歷程中使用時，所有其他事件型別也計入配額。

## 更新和刪除事件 {#update-event}


為避免中斷現有歷程，當您編輯用於&#x200B;**草稿**、**即時**&#x200B;或&#x200B;**已關閉**&#x200B;歷程的事件時，您只能變更名稱、說明或新增裝載欄位。

無法刪除&#x200B;**即時**、**草稿**&#x200B;或&#x200B;**已關閉**&#x200B;歷程中使用的任何事件。 要刪除已使用的事件，您必須停止使用該事件的歷程，和/或將其從使用它的草稿歷程中移除。 您可以檢查&#x200B;]**欄位中的**[!UICONTROL &#x200B;已使用。 它會顯示使用該特定事件的歷程次數。 您可以按一下&#x200B;**[!UICONTROL 檢視歷程]**&#x200B;按鈕以顯示對應歷程的清單。

## 常見問題 {#faq}

**我可以在多個歷程中使用相同事件嗎？**
是 — 數個歷程可同時聆聽相同的事件。

**我可以在相同歷程中結合業務事件和單一事件嗎？**
否 — 業務事件不能與單一事件或對象資格活動用於同一個歷程。

**我需要為對象資格事件設定任何專案嗎？**
否 — 未在**管理>事件**&#x200B;中設定對象資格事件。 直接在歷程畫布上選取對象作為第一個步驟。

**我可以使用批次擷取的資料來觸發歷程嗎？**
否 — 只有串流事件才能觸發歷程。 若要批次處理資料，請建置對象並改用**讀取對象**&#x200B;活動。

**我的歷程未觸發 — 我應該檢查什麼？**

* 確認您的事件結構描述已啟用&#x200B;**即時客戶個人檔案**。
* 確認事件已串流 — 批次擷取的事件無法觸發歷程。
* 對於規則型事件，確認規則條件符合傳入的裝載欄位。
* 檢查歷程是否處於&#x200B;**即時**&#x200B;狀態，以及設定檔是否符合任何進入條件。
